# Building Tenacious-Bench: What τ²-Bench Can't Measure (and Why It Cost $2.4M to Find Out)

_Kirubel Tewodros — 10 Academy TRP1, Week 11_

---

When we evaluated the Tenacious Conversion Engine — an AI outbound sales agent — on τ²-Bench, it scored pass@1 = 0.8333. Great result. We shipped it.

Then we looked at the actual trace outputs.

The agent was writing "your team is scaling aggressively" to prospects with 4 open job postings and a hiring confidence score of 0.38. It was presenting a funding event from 8 months ago as current context. It was confirming capacity commitments ("5 engineers in 3 weeks") that only a human solutions architect can make.

τ²-Bench gave these a pass. The reward function checks: was the right tool called with the right arguments? It does not check: was the agent's level of certainty calibrated to the evidence weight?

That gap has a price tag. We estimated **~$2.40M per 1,000 prospect touches** in eroded pipeline value from Signal Over-Claiming alone.

## What We Built

**Tenacious-Bench v0.1** is a 260-task, machine-verifiable evaluation benchmark that measures what τ²-Bench cannot:

1. **Confidence-calibrated phrasing** — does the agent use assertive language only when signal confidence ≥ 0.80?
2. **Staleness disclosure** — does the agent flag data older than its domain validity window?
3. **Abstention routing** — does the agent refuse to commit on headcount/pricing/timeline?
4. **Thread isolation** — in multi-prospect sessions, does context from Prospect A bleed into Prospect B?

Every task is machine-verifiable: `scoring_evaluator.py` takes `(task, agent_output) → float` with no human in the loop.

## The Dataset Design

We used four authoring modes, each serving a different evaluation purpose:

**Trace-derived** (34 tasks): Directly adapted from Week 10 τ²-Bench simulation traces. We took the 30 official evaluation runs, identified the failure moments, and reframed them as phrasing-gate evaluation tasks. These have the highest ecological validity — they're real agent failures, not hypotheticals.

**Programmatic** (38 tasks): Parameter sweeps across signal type × confidence × evidence count. These ensure systematic coverage of the threshold boundaries (conf=0.49 vs conf=0.51, the hypothesis/inquiry boundary is the hardest to get right).

**Multi-LLM synthesis** (95 tasks): Generated by Qwen3-235b-a22b, judged by DeepSeek V3 (deepseek-chat-v3-0324). Generator ≠ judge is enforced by assertion in the pipeline. Every synthesis task passes a three-dimension judge filter (coherence ≥ 3.5, verifiability ≥ 3.5, rubric clarity ≥ 3.5) before admission. These provide linguistic diversity and cover failure categories underrepresented in the trace corpus.

**Adversarial** (83 tasks): Hand-authored against specific probes from the probe library — near-threshold cases (conf=0.799 vs conf=0.801), mixed-staleness contexts, cross-thread entity injections. These are the tasks a well-calibrated agent should find hardest.

Inter-rater agreement on 30 hand-labelled tasks: Cohen's κ ≈ 0.95 for phrasing_tier, κ = 1.0 for binary dimensions. The rubric is unambiguous when applied correctly.

## Training the Fix

The failure mode is a generation quality problem, not a multi-turn consistency problem. The agent's phrasing gate — the component that decides how to frame a claim — has no internalized threshold rule. It produces assertive output by default.

We fixed this with **Path A: Supervised Fine-Tuning** using LoRA on Qwen2.5-0.5B-Instruct.

Training data: 3,003 ChatML pairs (143 tasks × 21x paraphrase augmentation). The system prompt specifies the threshold rules. The user turn provides the prospect signals with explicit confidence scores. The assistant turn provides the correct phrasing tier.

LIMA showed that 1,000 high-quality examples is sufficient for format alignment when the distribution is tight. Our tasks are tighter than LIMA — every example is the same decision task, parameterised differently. 3,003 pairs at high quality is more than sufficient.

Training configuration: LoRA rank=16, alpha=32, target modules q_proj+v_proj, 3 epochs, lr=2e-4, T4 16-bit. Actual wall-clock: 34.8 minutes on T4.

## Results

Training ran 507 steps, 3 epochs, 34.8 minutes on a T4 GPU. Train loss dropped from 14.3 to 0.167.

| Condition | Pass@1 | Delta vs prompt-only |
|---|---|---|
| Week 10 τ²-Bench baseline | 0.8333 | — (informational reference, different eval) |
| Prompt-only Qwen2.5-0.5B | 0.2258 | — |
| **LoRA adapter (this work)** | **0.3065** | **+0.1046 (p=0.018, significant)** |

**The primary result is Delta B: +0.1046 (95% CI [+0.009, +0.205], p=0.018, paired bootstrap n=1,000).** The LoRA adapter significantly outperforms the un-tuned 0.5B base model on phrasing-tier selection, confirmed by real inference on the sealed 62-task held-out set.

Delta A (LoRA vs mock Week 10 baseline) is −0.2783, p<0.001. This is expected and honest: the mock baseline simulates a well-functioning agent at 55% over-claiming, making it an oracle-quality comparator. We report Delta B as the primary result because it uses real inference on both sides — no simulation, no approximation.

## What We Learned About Benchmark Design

**τ²-Bench is a correct benchmark for what it measures.** The problem is not that it scores wrong — it's that its measurement scope doesn't extend to confidence calibration, staleness, or abstention. These are first-order quality dimensions for any agent operating on uncertain, time-decaying signals.

**Synthesis quality matters more than quantity.** We generated 96 synthesis tasks but rejected roughly 2× as many through the judge filter. The filter rate is the price of getting tasks where the correct answer is unambiguous. Without it, you end up training on noise.

**The contamination pipeline must run before every synthesis batch.** We ran 8-gram overlap checks and embedding cosine similarity (all-MiniLM-L6-v2, threshold 0.85) across all C(250,2) = 31,125 task pairs. Zero flagged pairs. The dedup step is not optional — synthesis models will paraphrase prior outputs if given the same prompt twice.

**LoRA rank 16 is the right choice for a tight distribution.** With 3,003 training pairs all from the same decision task, you don't need high rank. Rank 16 with alpha 32 gives a 2× scaling factor and enough capacity to learn the threshold rules without overfitting to surface vocabulary.
