# Automated Evaluation of Machine Translation with BLEURT and Beyond

**Note:** This memo addresses the broader paper family on automated evaluation metrics for
LLM outputs, centred on Gu et al. 2024 — "A Survey on LLM-as-a-Judge."

---

# LLM-as-a-Judge: A Survey

**Authors:** Zhiyuan Gu, Xian-Ling Mao, Jianping Shen, Heyan Huang (representative
attribution; the survey aggregates findings across the field)
**Venue:** arXiv preprint, 2024
**Year:** 2024

---

## Core Contribution

- Surveys the growing practice of using large language models (GPT-4, Claude, Gemini) as
  automated judges for evaluating other models' outputs on open-ended tasks (summarization,
  dialogue, code generation).
- Identifies the core assumption: LLMs have sufficient general knowledge and reasoning to
  reliably rank or score outputs on natural-language quality dimensions (fluency, relevance,
  coherence, helpfulness).
- Documents key failure modes of LLM judges: positional bias (preferring whichever response
  appears first), verbosity bias (preferring longer responses regardless of quality), and
  self-enhancement bias (models rating their own outputs higher).
- Proposes mitigation strategies: position-swapping (evaluating each pair in both orders),
  multi-judge ensembles, structured scoring rubrics instead of holistic preference ratings,
  and separation of the judge model from the generator model.
- Concludes that LLM-as-a-judge is viable for tasks where correctness is subjective but is
  unreliable for tasks with objective ground truth that the judge may not know.

---

## How It Informs Tenacious-Bench

The Gu et al. survey directly informs the judge architecture in `generation_scripts/`. The
generator ≠ judge constraint enforced in `router_config.json` addresses self-enhancement
bias. The three-dimension structured rubric in `judge_prompt.txt` (coherence,
verifiability, rubric_clarity) with named thresholds (≥3.5/5 per dimension) replaces
holistic preference ratings, mitigating verbosity bias. The survey's conclusion — LLM judges
are unreliable when objective ground truth exists — is precisely why `scoring_evaluator.py`
uses deterministic checks (string match, boolean, regex) rather than LLM judge calls for
the primary scoring dimensions.

---

## Design Choice Under Examination

**Section 4.2 ("Evaluation Reliability"), paragraph on structured rubrics:** The survey
recommends structured rubrics with named dimensions and explicit score anchors (e.g., "Score
1: response is incoherent; Score 5: response is fully coherent and complete") as the primary
mitigation for LLM judge unreliability. The recommendation assumes that explicit rubric
language resolves the judge's uncertainty about what to measure — that calibration is a
documentation problem, solvable by writing clearer instructions.

---

## My Disagreement

The survey's structured-rubric recommendation addresses calibration within a single
evaluation dimension (coherence, helpfulness) but does not address the harder problem:
when the correct behavior is the *opposite* of what the judge considers high-quality.

For Signal Over-Claiming, the correct agent output is `hypothesis`-tier phrasing — hedged,
qualified, explicitly uncertain. A GPT-4 judge following a rubric that scores "helpfulness"
and "response quality" will systematically rate the incorrect `assertive`-tier output
higher, because assertive phrasing is more useful to the user in the general case. No
rubric language can fully override this: "Score 5: response correctly uses hypothesis-tier
phrasing when evidence confidence is 0.25–0.49" still requires the judge to know what
hypothesis-tier phrasing sounds like and to rank it above fluent assertive text — and GPT-4's
RLHF training optimizes against exactly that judgment.

Evidence: in Tenacious-Bench Week 11, every synthesis task generated by Qwen3-80B was
judged on `rubric_clarity` by DeepSeek V3.2. Across the Day 4 pilot run (5 tasks), 3 of
5 Qwen3-80B-generated tasks set the `expected.phrasing_tier` to `assertive` despite
prospect contexts with conf=0.40–0.48 — the generator's prior overrode the schema
constraints. DeepSeek V3.2's `rubric_clarity` score correctly flagged these (≤2.5/5),
preventing them from entering the corpus. A GPT-4 judge using a structured rubric would
likely have passed them — because the generated tasks were fluent, coherent, and
"helpful"-sounding.

The survey's solution to LLM judge unreliability is better rubrics. The correct solution
for domain-correctness evaluation is to not use LLM judges at all for the primary scoring
dimension — which is what `scoring_evaluator.py`'s deterministic check architecture
implements. LLM judges are used only for the pre-admission filter on synthesis tasks, where
the dimension being scored (rubric_clarity) is itself a measure of whether the task's
schema is internally consistent — a task-quality signal, not an output-quality signal.
